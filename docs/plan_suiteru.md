# 「すいてる東京」設計計画書

公共交通オープンデータチャレンジ2026 応募作品(norishiroとは独立の第2作品)

- 作成日: 2026-07-16
- 対象読者: 実装を担当するAIセッションと、指示を出す開発者本人(プログラミング初心者・pandas経験あり)
- この文書の使い方: norishiroの plan_gap_map.md と同じ流儀。マイルストーン(M0〜M8)を
  1つずつ実装セッションに渡し、「完成条件」を満たしてから次に進む。
- **応募締切: 2027年1月11日(月・祝)**(応募期間は2026年10月1日〜)。
  norishiroが主作品なので、本作品は余力で進める。**M5まで(Stage 1)で応募可能な完成品**になる構成。

## 1. コンセプト

> 「いつ行くか」を変えるだけで、東京はすいてる。

訪日外国人観光客に、**混雑予報**に基づいて行程の**時間と順序**を組み替えた観光プランを
提案するWebサービス。テーマは2026年重点課題の**オーバーツーリズム対策**。

- 行き先を変えさせる(我慢させる)のではなく、**同じ場所を良い時間に**案内する
- リアルタイム混雑データはオープンデータに存在しない(2026-07-16調査済み)。
  よって**「統計に基づく予報」と正直に名乗る**。リアルタイム要素は運行情報(遅延・運休)のみ
- UIは日英2言語。対象は混雑が社会問題化している主要観光スポット15〜30箇所

## 2. 2段構え(最重要の設計方針)

**Stage 1(M0〜M5)= 静的サイトのみで完結する応募可能な完成品。**
**Stage 2(M6〜M8)= Worker+リアルタイム+LLMコンシェルジュ(余力があれば)。**

後付けを成立させる設計ルール:

1. 行程最適化は最初から**入出力JSON固定の純粋なJS関数** `plan_itinerary(request) → itinerary`
   として書く。Stage 1では画面のフォームが、Stage 2ではLLMが、同じ関数を呼ぶだけ
2. UIの主動線は**手動の行程ビルダー**として完成させる。チャットは後から足す別の入口
3. APIキーが必要な要素(ODPT運行情報・Claude API)はすべてStage 2のCloudflare Workerに隔離。
   Stage 1は事前計算済みJSONだけで動き、サーバー運用ゼロ(GitHub Pages)
4. Worker導入は「運行情報の中継」(単純プロキシ)から始め、慣れてからLLMを載せる

```
[混雑統計データ] ─┐
[観光スポット情報] ├─ Python事前計算 ─→ 静的JSON ─→ 静的Webサイト(GitHub Pages)
[GTFS(東京の鉄道)] ┘  (precompute/)      (webapp/data/)   ・「すいてる時間」ヒートマップ
                                                          ・行程ビルダー(JSで完結)
                                          --- Stage 2 ---
                                          Cloudflare Worker(無料枠)
                                          ・ODPT運行情報の中継(M6)
                                          ・LLMチャット(M7): 意図抽出と説明文生成のみ。
                                            行程計算はM4の関数を呼ぶ(幻覚対策)
```

## 3. データ設計(M0で実在確認)

混雑プロファイルの材料は確度順に以下を当たり、最初に成立したものを採用して本章に追記する:

1. **大都市交通センサス(国交省、2021年調査)** — 駅別・時間帯別利用者数。
   スポットの最寄り駅の時間帯カーブを混雑プロキシにする(第一候補)
2. **スムーズビズ関連の駅混雑データ**(東京都/東京メトロの時間帯別混雑) — 存在と再配布可否を確認
3. **全国の人流オープンデータ**(国交省、1kmメッシュ、2019〜2021年) — 平休日×昼/夜の粗い粒度。
   コロナ期データのため採用時は2019年基準+限界明記
4. 自治体の人流調査(台東区の浅草調査等) — スポット個別の補正材料

その他のデータ:

- **観光スポット**: 東京都オープンデータカタログの多言語観光データを確認。
  なければ手動整備(15〜30箇所ならCSV手作りで十分)
- **経路**: ODPTセンターの鉄道GTFS(JR東・メトロ・都営等)。
  **norishiroの自作RAPTORエンジン(gap_map/transit_core.py)を移植して再利用**
- **運行情報(Stage 2)**: ODPT APIリアルタイム運行情報(要開発者登録)

## 5. マイルストーン

### M0: データ実在確認とプロジェクト骨格(最初の関門)
- §3のデータ候補1〜4を実際に取得して評価し、混雑プロファイルの採用ソースを
  **根拠つきで docs/data_survey.md に記録**。ODPT開発者登録(人間の作業)
- 完成条件: 採用ソースが文書化され、サンプルデータが data/ にある

### M1: スポットマスタ(15〜30箇所)
- `data/spots.csv`: id, 名称(ja/en), 緯度経度, 最寄り駅(複数可), カテゴリ, 営業時間,
  混雑が社会問題化している根拠メモ
- 完成条件: 浅草寺・渋谷スクランブル・築地場外・明治神宮等を含む20件以上

### M2: 混雑プロファイル事前計算
- スポット×曜日タイプ×時間帯(1時間刻み)の混雑スコア(0〜100)
  → `webapp/data/crowd_profiles.json`
- 完成条件: 全スポットの24時間カーブが既知の事実(築地=午前ピーク、渋谷=夕方以降など)と
  矛盾せず、根拠データへのトレースを説明できる

### M3: スポット間所要時間行列
- transit_core.py移植+東京の鉄道GTFSで スポット×スポット所要時間
  → `webapp/data/travel_matrix.json`
- 完成条件: 抜き取り5ペアをNAVITIME等と照合し±15分以内

### M4: 行程最適化エンジン(ルールベース・JS)
- 入出力JSON固定の純粋関数。混雑スコア合計を最小化する訪問順序と時間割
  (スポット数≤6なら全順列探索で足りる)。「通常プラン比で混雑度○%減」の比較を出力に含める
- 完成条件: 単体テスト+手動ケース(浅草+渋谷+築地の1日プランで築地が朝に来る)

### M5: 静的Webフロント(日英) — Stage 1のゴール
- スポット一覧+「すいてる時間」ヒートマップ+手動行程ビルダー。GitHub Pages公開
- 完成条件: スマホ幅で崩れず、英語UIだけで一連の操作が完結。
  **README・既知の限界の明記まで揃え、この時点で応募可能な状態にする**

## 6. 既知のリスクと対応

| リスク | 対応 |
|---|---|
| 時間帯別の混雑データが見つからない | M0を関門にし、最悪は人流データ(昼/夜2区分)+公表調査の引用で「粗い予報」として成立させる。プロファイルJSONの差し替えだけで済む分離設計 |
| Worker運用が初心者に重い | Stage 2を丸ごと任意化。M5までで応募可能 |
| norishiroとの並行で時間切れ | M0〜M5はnorishiroと同じ技術(Python+静的サイト)。Stage 2は割り切る |
| Claude API費用 | Haiku 4.5+トークン上限+コンソールで支出上限設定 |

## 7. 既知の限界(READMEにも書く)

1. 混雑は**統計に基づく予報**であり実測ではない(データ年次も明記する)
2. 対象は登録済みスポットのみ
3. 人流データの時間帯区分は「昼(11-14時台)/深夜(1-4時台)/終日」のみ。
   1時間刻みのカーブは実測でなくタイプ別テンプレート(curve_templates.json・出典つき)が担う
4. **1kmメッシュの粗さ**: スポットの混雑レベルは周辺1kmメッシュ全体の滞在人口を使うため、
   繁華街や駅に隣接するスポットは実際より高く出る(M2で確認した例: 新宿御苑は新宿三丁目を、
   谷中銀座は日暮里駅を含むメッシュのため過大評価気味。上野公園とアメ横は同一メッシュ)
5. レベルの基準年は2019年(コロナ前)。訪日客構成は現在と異なる
